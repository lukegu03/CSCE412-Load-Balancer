Description

This page documents the workflow for my CSCE 412 Load Balancer simulation. First, the request is created with random IP_in, IP_out, job type (P/S), and processing time. The request is then validated by the firewall before being sent to the load balancer. The load balancer queues the requests. When a server is available, the next request in the queue is dispatched to the server for processing. The server handler is responsible sending the request to an available server. Once the request has been processed, the server is marked as available again. After a time interval, n, the system checks the queue length. If the queue length is above 80*servers, a new server is added. If the queue length is below 50*servers, a server is removed.

High-level workflow:

  • Request generation: Random requests are created with IP_in, IP_out, job type (P/S), and processing time.
  • Firewall validation: Requests from blocked IP ranges are dropped and logged. Requests that are approved are sent to the load balancer.
  • Load Balancer: Valid requests are pushed into the load balancer queue.
  • Dispatch: When a server is available, the next queued request is assigned to a server to be processed.
  • Dynamic Server allocation: The system will scale up or down depending on the number of requests in the queue and the current server utilization. If the queue goes below 50*servers the system will scale down. If the queue goes above 80*servers, the system will scale up.
  • Logging: Important actions (blocked requests, server add/remove, dispatch) are recorded for summary output.
